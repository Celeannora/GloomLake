# 10-Mythic Synergy Framework — Implementation Plan

## Mission
Rebuild the synergy analysis system as a unified, mathematically rigorous framework modeled on 10 distinct Mythic-level MTG player archetypes. The output must be suitable as input to autoresearch/optimization loops (Karpathy-style).

---

## Phase 1 — 10 Mythic Player Archetypes

Each archetype defines a distinct optimization lens over the same deck configuration. Together they form the panel that scores any deck from 10 independent strategic perspectives.

| # | Archetype | Strategic Identity | Primary Optimization Axis |
|---|-----------|-------------------|--------------------------|
| 1 | **Spike** | Pure competitive optimization | EV maximization per mana |
| 2 | **BrewerPro** | Synergy web density | Highest pairwise interaction count |
| 3 | **Grinder** | Attrition / card advantage | Resource efficiency over time |
| 4 | **AggroPro** | Tempo / early pressure | Damage-per-turn curve |
| 5 | **ControlMaster** | Inevitability | Late-game lock & answer density |
| 6 | **ComboArchitect** | Critical mass / combo windows | Minimum piece count threshold |
| 7 | **TribalChief** | Tribal synergy density | Tribe-internal interaction amplification |
| 8 | **RampSage** | Resource acceleration | Mana-ahead delta per turn |
| 9 | **GraveyardGuru** | Recursion / value loops | Graveyard-to-battlefield efficiency |
| 10 | **MetaHunter** | Meta-specific hate/positioning | Expected meta matchup win rate |

### 2.2 Synergy Coefficient (SC)

Measures how well a card integrates into the deck's synergy web:

```
SC(card, deck) =
    (engine_density   * 40.0)   +   # engine_partners / engine_pool_size
    (synergy_density  * 25.0)   +   # synergy_partners / pool_size
    min(weighted_synergy, 20.0) +   # Σ interaction_weights * confidence
    (role_breadth     * 3.0)    +   # distinct InteractionType count
    (oracle_confirmed * 2.0)        # oracle-text verified links
```

**FIX #1:** Wire `dependency` into SC as a penalty:
```
SC_adjusted = SC(card, deck) - (dependency_score * 2.5)
```
Dependency penalizes cards that require other cards to function (e.g., auras without enchant targets).

**FIX #2:** Wire oracle cross-reference into weighted_synergy:
```
oracle_bonus = confidence_multiplier["oracle"] * base_weight
# Currently 0, should be 1.5 * interaction_weight
```

### 2.3 Expected Value (EV) of a Deck Configuration

```
EV(deck) = Σ_cards [ SC_adjusted(card) * qty_weight(card) ] / N
         + axis_coherence_bonus(deck)
         - isolation_penalty(deck)
         - dependency_fragility(deck)

where:
  qty_weight(card)          = 1 + 0.15 * (qty - 1)         # Copies increase reliability
  axis_coherence_bonus      = primary_axis_concentration^2 * 10.0
  isolation_penalty         = isolated_fraction * 15.0
  dependency_fragility      = avg_dependency_score * 5.0
```

### 2.5 Mana Curve Score

```
curve_score(deck) =
    1.0 - |avg_cmc(deck) - target_cmc(archetype)| / 5.0

target_cmc = {
  Aggro: 1.8, BrewerPro: 2.5, Spike: 2.4, Grinder: 3.0,
  Control: 3.2, Combo: 2.8, Tribal: 2.2, Ramp: 3.5,
  Graveyard: 2.6, Meta: 2.6
}

# Curve shape penalty: punish top-heavy decks
shape_penalty = max(0, (cmc_4plus_fraction - 0.3)) * 10.0
curve_score_adjusted = curve_score - shape_penalty
```

### 2.6 Resource Bottleneck Detection

```python
def detect_bottleneck(deck) -> dict:
    return {
        "mana_bottleneck":    land_count < (avg_cmc * 7),
        "draw_bottleneck":    draw_engine_count < 4,
        "removal_bottleneck": removal_count < 8,
        "payoff_bottleneck":  payoff_count < 6,
        "engine_bottleneck":  engine_count < 4,
        "tribe_bottleneck":   tribe and tribe_count < 20,
    }
```

### 2.7 Panel Consensus Score (Autoresearch Output)

```
panel_score(deck) = {
    "per_archetype": {name: fitness(deck, archetype) for archetype in PANEL},
    "consensus":     mean(fitness values),
    "variance":      std(fitness values),      # Low variance = universally good
    "bottlenecks":   detect_bottleneck(deck),
    "recommendations": top_k_cuts_and_adds(deck, pool, k=5),
    "ev":            EV(deck),
}
```

**Variance insight:** Low variance = strong, well-rounded deck. High variance = polarizing build (good for some archetypes, bad for others).

## Phase 3 — Implementation Plan

### Decision: Architecture Choice
After analysis, the cleanest approach is:
- **Replace** `mythic_panel.py` with `mythic_framework.py` (old file deleted)
- **Wire** into `synergy_analysis.py` via `--mythic-panel` flag
- **Append** panel scores to the existing `synergy_report.md` AND produce a companion `panel_report.json`

This keeps a single entrypoint, one report file, and a machine-readable JSON for autoresearch loops.

### New File: `scripts/analysis/mythic_framework.py`

Single authoritative module that replaces/extends `mythic_panel.py` with:

1. **`ArchetypeProfile` dataclass** — holds weights vector, target_cmc, archetype-specific thresholds
2. **`MYTHIC_PANEL` constant** — list of 10 `ArchetypeProfile` instances
3. **`compute_ev(scores: Dict[str, CardScore]) -> float`** — deck EV formula
4. **`fitness(scores, archetype) -> float`** — per-archetype fitness
5. **`detect_bottlenecks(scores) -> dict`** — resource bottleneck map
6. **`run_panel(scores, tribe=None) -> dict`** — full 10-player panel output
7. **`recommend_cuts_adds(scores, pool_scores, k=5) -> list`** — top-k recommendations from pool

# Restructuring Plan for GloomLake Repository

## Goal
Consolidate loose files and organize the repository for better maintainability.

## Current Issues
- Too many loose files in the root directory (e.g., wrong.png, various .md files, .json files).
- Scripts and other files are scattered without clear organization.
- Documentation is spread out (docs/ vs root vs plans/Archive/).
- Asset and data folders are not clearly separated.

## Proposed Solution

### 1. Directory Structure
We propose the following structure:

```
GloomLake/
├── assets/                     # All static assets (SVGs, icons, etc.)
│   ├── svg/                   # SVG files
│   ├── data/                  # Game data (card database, etc.)
│   │   └── cards_by_category/ # Card data CSV files
│   └── templates/             # Template files (deck templates, etc.)
├── docs/                      # All documentation
│   ├── getting-started/       # Guides for new users
│   ├── reference/             # Detailed references (AI instructions, etc.)
│   └── changelog/             # Changelog and specifications
├── scripts/                   # All Python scripts (code)
│   ├── analysis/              # Synergy analysis and related
│   ├── utils/                 # Utility functions
│   └── cli/                   # Command-line interface scripts
├── tests/                     # Unit and integration tests
├── Decks/                     # Generated decks (gitignored)
├── local_db/                  # Local database (gitignored)
├── test-corpus/               # Test data for development
├── [config files]             # pyproject.toml, requirements.txt, etc.
├── [GUI entry point]          # scaffold_gui.py (may remain at root for convenience)
```

### 2. File Movements

#### Root Level
- Move all .md files (except those already in docs/) to appropriate docs/ subdirectories:
  - README.md -> docs/getting-started/README.md
  - LOCAL_WORKFLOW.md -> docs/getting-started/LOCAL_WORKFLOW.md
  - AI_INSTRUCTIONS.md -> docs/reference/AI_INSTRUCTIONS.md
  - Changelog.md -> docs/changelog/CHANGELOG.md
  - SPEC.md -> docs/changelog/SPEC.md
  - Any other .md files in root -> appropriate docs/ subfolder
- Move wrong.png to assets/svg/ or delete if unnecessary.
- Move any loose .json files to appropriate locations (e.g., test-corpus/ or tests/fixtures/).

#### Scripts Organization
- Organize scripts/ into subdirectories by function:
  - Move synergy-related scripts to scripts/analysis/
  - Move utility scripts to scripts/utils/
  - Keep top-level scripts (like generate_deck_scaffold.py, search_cards.py) in scripts/ or move to scripts/cli/
- Update import paths in scripts to reflect new structure.

#### Test Organization
- Move test-corpus/ contents to tests/fixtures/ for consistency.
- Keep tests/ for test scripts.

### 3. Necessary Code Changes
- Update paths in scripts that reference moved files (e.g., in mtg_utils.py for card data paths).
- Update any hardcoded paths in the GUI (scaffold_gui.py) for assets and templates.
- Ensure all relative paths are correct after moving files.

### 4. Verification Steps
- After moving files, verify that the GUI still launches and functions correctly.
- Run a few key scripts (search_cards.py, validate_decklist.py, generate_deck_scaffold.py) to ensure they work.
- Run the test suite to ensure nothing is broken.

### 5. Phased Implementation

#### Phase 1: Backup and Preparation
- Create a backup of the current state (optional but recommended).
- Identify all files to be moved and their target locations.

#### Phase 2: Directory Creation
- Create all necessary directories in the new structure.

#### Phase 3: File Movement
- Move files to their new locations, starting with documentation and assets.

#### Phase 4: Code Updates
- Update all path references in Python scripts and GUI.

#### Phase 5: Verification
- Test the GUI and key scripts.
- Run tests to ensure functionality is preserved.

## Benefits
- Improved organization and maintainability.
- Clear separation of concerns (assets, docs, code, tests).
- Easier navigation for new contributors.
- Reduced clutter in the root directory.

## Risks and Mitigation
- Risk: Breaking scripts due to incorrect path updates.
  Mitigation: Update paths carefully and test each script after changes.
- Risk: Losing track of file movements.
  Mitigation: Keep a detailed list of all movements and verify after each phase.

## Conclusion
This restructuring will make the repository more manageable and professional, facilitating future development and collaboration.
